## Folder structure

All projects use this structure. Claude has permission to create new files in any of these folders.

```
project/
├── src/
│   ├── index.html
│   └── assets/
│       ├── css/
│       ├── js/
│       └── images/
├── dist/               # generated by Vite — mirrors src/ structure
├── screenshots/        # generated at delivery
├── docs/
│   ├── brand_assets/   # source of truth for logo, fonts, imagery
├── vite.config.js
└── package.json
```

HTML files live in `src/`. CSS, JS, and images each have their own subfolder under `src/assets/`. For multi-page portfolios, each page has its own `.html` file in `src/` (e.g. `src/about.html`, `src/projects.html`). Never mix file types or place assets in the project root.

## Environment setup

This workflow is for **plain HTML/CSS/JS with Vite** — no frameworks, but npm packages are available via the build step.

**Starting a new project**
```bash
npm create vite@latest . -- --template vanilla
npm install
```

Then replace the default `vite.config.js` with:
```js
import { defineConfig } from 'vite'

export default defineConfig({
  root: 'src',
  build: {
    outDir: '../dist',
    emptyOutDir: true,
  },
})
```

Create the folder structure:
```bash
mkdir -p src/assets/css src/assets/js src/assets/images screenshots
```

Then start the dev server:
```bash
npm run dev
```
Default dev URL: `http://localhost:5173`

**Adding to an existing project**
```bash
npm init -y
npm install --save-dev vite
```
Add to `package.json`:
```json
"scripts": {
  "dev": "vite",
  "build": "vite build",
  "preview": "vite preview"
}
```
Add the `vite.config.js` above, create the folder structure, then run `npm run dev`.

**Installing dependencies**
```bash
npm install gsap        # example: animation library
npm install swiper      # example: slider
```
Import in JS using ES modules — do not use CDN links:
```js
import gsap from 'gsap'
import Swiper from 'swiper'
```

Brand assets (logo, fonts, images, brand guide) are always located in the `brand_assets/` folder. Always check this folder before making assumptions about colours, typography, or imagery.

## Portfolio website workflow

### 1. Clarify before building

Before starting, confirm:
- What is the primary goal? (get hired, attract freelance clients, showcase creative work, all three)
- What is the person's role/discipline? (developer, designer, developer-designer, creative technologist, etc.)
- Who is the target audience? (recruiters, agencies, direct clients, peers)
- Are there brand assets, colors, or fonts to use? Check the `brand_assets/` folder first — brand files are always stored there.
- Is there existing copy (bio, project descriptions), or does it need to be generated?
- Which projects should be featured, and in what order of priority?
- What skills and tools should be listed?
- Should there be a blog or case study section?
- Which language should the copy be in?
- Is a contact form required, or is an email link sufficient?

---

### 2. Define the page structure

Plan the section order before coding. A portfolio is typically a single-page layout with anchor navigation, or a multi-page site with a persistent nav.

| Section | Purpose |
|---|---|
| Hero | First impression — name, role, and one-line value statement |
| About / bio | Human context — who you are, what drives you, where you're based |
| Skills | Scannable list of tools and technologies |
| Projects / work | The core — show what you've built, not just what you know |
| Experience (optional) | Timeline of roles, clients, or education |
| Testimonials (optional) | Social proof from clients or colleagues |
| Contact | Clear path to reach out — form, email, or both |

For multi-page sites, also plan:
- Individual project/case study pages with problem → process → outcome structure
- A dedicated page about accessibility
- A dedicated about page if the bio section is substantial

---

### 3. Apply the frontend-design skill

Follow the skill's design thinking steps:

- **Purpose**: Define the problem this portfolio solves and who it targets
- **Tone**: Commit to a clear aesthetic that reflects the owner's discipline — a frontend developer's portfolio should feel polished and technical; a creative's should feel expressive
- **Differentiation**: Identify the one memorable design choice that makes this portfolio non-generic
- **Typography**: Choose a distinctive font pair — a strong display face for name/headlines, a readable body face for bios and descriptions
- **Color**: Pick a dominant palette with a sharp accent; use CSS variables throughout; dark mode is a strong default for dev portfolios
- **Motion**: Plan one well-orchestrated page load with staggered name/role reveal; add subtle scroll-triggered project card entrances and hover lift states
- **Layout**: Use an intentional grid — asymmetric hero, featured project cards at varying sizes, skills in a clean inline or grid layout

---

### 4. Copywriting principles for portfolios

- The hero headline is the name + role, not a tagline — keep it clear and confident
- The bio should be first-person, specific, and short (3–5 sentences max)
- Project descriptions follow: what it is → what problem it solved → your specific contribution → outcome or result
- Skills section should list tools and technologies without fluff — no "proficient in" or "experienced with"
- CTA copy should be direct: "View project", "See case study", "Get in touch", "Download CV"
- Avoid generic phrases: "passionate about", "problem solver", "detail-oriented", "team player"
- Use specific numbers where possible: "Built X for Y clients", "Reduced load time by Z%"

---

### 5. Projects section guidelines

This is the most important section. Apply these rules:

- Show 3–6 projects maximum; quality over quantity
- Each card must include: project name, one-line description, tech stack tags, and a link or "view case study" CTA
- Feature the strongest project first — it sets the bar
- Use real screenshots or mockups; avoid placeholder images
- If linking to live projects, open in a new tab with `target="_blank" rel="noopener noreferrer"`
- Case study pages (if present) follow: brief → problem → process → solution → outcome
- Include the owner's specific role on each project — "Designed and built", "Frontend development", "Shopify theme customization", etc.

---

### 6. Skills section guidelines

- Group skills by category: Languages, Frameworks, Tools, Platforms
- Use a clean inline or tag-cloud layout — avoid long prose lists
- Do not rate skills with bars or percentages — these are subjective and add no signal
- Only list skills that are genuinely usable — not things learned once in a tutorial

### 7. Delivery automation

Run these commands after the build is complete.

**Build for production**
```bash
npm run build
```
Vite outputs minified HTML, CSS, and JS to `dist/` automatically. No separate minification step needed.

**Preview the production build**
```bash
npm run preview
```
Default preview URL: `http://localhost:4173`

**Accessibility audit** (run against the preview server)
```bash
npx pa11y http://localhost:4173 --standard WCAG2AA
```

**Performance, SEO & best practices (Lighthouse)**
```bash
npx lighthouse http://localhost:4173 --output html --output-path ./lighthouse-report.html --chrome-flags="--headless"
```

**Screenshots for feedback**

For a single-page portfolio:
```bash
mkdir -p screenshots
npx capture-website-cli http://localhost:4173 --output screenshots/desktop.png --width 1440 --height 900 --full-page
npx capture-website-cli http://localhost:4173 --output screenshots/tablet.png --width 768 --height 1024 --full-page
npx capture-website-cli http://localhost:4173 --output screenshots/mobile.png --width 375 --height 812 --full-page
```

For a multi-page portfolio, repeat for each page using matching filenames — e.g. `screenshots/about-desktop.png`.

**Package for handoff**
```bash
zip -r delivery.zip dist/ screenshots/
```

> If pa11y or Lighthouse flag issues, fix them before packaging. Do not deliver with known WCAG AA violations.
